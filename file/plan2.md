# 计划 2：AI 生成 3D 桌宠模型管线

## 目标

实现一个桌宠系统：用户可以上传人物或宠物照片，生成风格统一的 3D 桌宠模型，然后在 Godot 游戏里执行固定动作，例如待机、挥手、跳跃、开心反应、点击反应，并且后续可以扩展手持道具、帽子、背包等装扮内容。

用户不需要去外部 AI 网站操作。AI 生成可以通过你的后端服务完成，Godot 客户端主要负责桌宠互动、动作播放、装扮挂载、本地缓存和游戏体验。

## 产品方向

采用两层架构：

1. 标准角色系统
   - Godot 游戏只加载符合固定规范的 3D 模型。
   - 规范包括骨骼命名、动画名称、模型尺寸、朝向、材质限制、装扮挂点等。
   - 这样输入触发动作用起来稳定，后续增加装扮也不会乱。

2. AI 生成服务
   - 后端接收用户上传的照片。
   - 后端调用第三方 AI 3D 生成服务。
   - 后端把生成结果整理成你的标准角色格式。
   - 后端返回一个验证通过的 `.glb` 文件给 Godot 游戏。

推荐的用户体验：

```text
用户在游戏内上传照片
-> 游戏创建生成任务
-> 后端生成并验证模型
-> 游戏下载 GLB
-> 模型缓存在本地
-> 桌宠可以执行标准动作和使用装扮
```

## 第一版建议

不要一开始就追求“任意照片都能完美生成可动模型”。第一版应该先做受控、可靠的版本：

- 输出风格：Q版、玩具感、简化 3D 角色。
- 支持主体：一张图里只支持一个人或一只宠物。
- 输入要求：主体清楚，背景简单，最好是全身照。
- 支持动作：待机、挥手、跳跃、开心、点击反应。
- 支持装扮挂点：右手。
- 支持模型格式：`.glb`。

这样能明显降低失败率，也更容易做成稳定的产品体验。

## Godot 运行时规范

所有被游戏接受的生成模型，都应该满足下面的规则。

### 模型格式

- 文件类型：`.glb`
- 尺寸：统一缩放到桌宠合适大小
- 朝向：保持一致，建议面向 `-Z`
- 地面位置：脚底或身体底部接近原点
- 材质：简单 PBR 或卡通材质
- 贴图大小：限制最大尺寸，避免性能问题

### 骨骼要求

人形/Q版角色建议使用下面的标准骨骼：

```text
Hips
Spine
Chest
Neck
Head
LeftUpperArm
LeftLowerArm
LeftHand
RightUpperArm
RightLowerArm
RightHand
LeftUpperLeg
LeftLowerLeg
LeftFoot
RightUpperLeg
RightLowerLeg
RightFoot
```

宠物/四足动物建议使用单独的标准骨骼：

```text
Root
Body
Chest
Neck
Head
Tail
FrontLeftUpperLeg
FrontLeftLowerLeg
FrontLeftFoot
FrontRightUpperLeg
FrontRightLowerLeg
FrontRightFoot
BackLeftUpperLeg
BackLeftLowerLeg
BackLeftFoot
BackRightUpperLeg
BackRightLowerLeg
BackRightFoot
```

第一版不要混用人形动作和四足动物动作。最好把它们当成两类模型，分别使用不同的动画库。

### 必需动画

人形角色的最低动画集：

```text
idle
wave
jump
happy
tap_react
```

宠物角色的最低动画集：

```text
idle
hop
happy
tap_react
sleep
```

如果某个生成模型不能稳定支持骨骼动画，可以使用降级动画：

- 整体上下漂浮
- 整体轻微旋转
- 整体缩放弹跳
- 如果有头部骨骼，就做轻微转头

### 装扮挂点

建议先定义这些挂点：

```text
RightHandSlot
HeadSlot
BackSlot
```

在 Godot 里可以用 `BoneAttachment3D` 实现。

第一版只需要做 `RightHandSlot`，因为手持道具最贴近你的核心玩法扩展方向。

## 后端服务计划

创建一个后端服务，把第三方 AI 服务细节藏起来，不让 Godot 客户端直接暴露 API Key。

### 核心接口

```text
POST /api/pets/generate
```

创建生成任务。

输入：

- 用户图片
- 模型类型：`humanoid` 或 `pet`
- 风格预设

输出：

- 任务 id

```text
GET /api/pets/generate/{task_id}
```

查询生成状态。

状态建议：

- queued：排队中
- generating：生成中
- rigging：绑定骨骼中
- validating：验证中
- complete：完成
- failed：失败

```text
GET /api/pets/models/{model_id}.glb
```

下载最终验证通过的模型。

### 后端职责

- 临时保存用户上传图片。
- 调用 AI 3D 生成服务。
- 要求输出风格化、适合动画的模型。
- 人形模型尽量要求 A-Pose 或 T-Pose。
- 尽量要求四肢分离、结构清晰。
- 下载第三方生成结果。
- 转换或整理成 `.glb`。
- 验证骨骼和动画是否满足要求。
- 必要时添加模型元数据。
- 保存最终模型。
- 把模型 id 返回给客户端。

### 可测试的 AI 服务

优先测试：

- Meshy
- Tripo AI
- Hyper3D / Rodin

选择服务时重点比较：

- 图片转 3D 的效果
- 自动骨骼绑定能力
- API 是否好接
- 是否支持导出 `.glb`
- 商用条款
- 单次生成成本
- 生成耗时
- 失败率

## 模型验证管线

生成模型不能直接信任，必须做验证。

验证清单：

- GLB 可以正常加载。
- 文件大小在限制以内。
- Mesh 数量合理。
- 贴图数量和尺寸合理。
- 存在骨骼。
- 必需骨骼存在，或者可以映射到标准骨骼。
- 模型朝向正确。
- 模型尺寸正确。
- 必需动画存在，或者可以套用标准动画。
- 可以创建装扮挂点。
- 播放动画时，模型包围盒没有异常拉伸。

如果验证失败，给用户友好的提示：

```text
这张图片暂时无法生成稳定的可动桌宠。请尝试上传背景简单、主体清晰的全身照片。
```

## Godot 客户端计划

### 阶段 1：本地标准角色系统

先不要接 AI。先用一个质量稳定的 GLB 模型，把本地桌宠玩法跑通。

任务：

- 导入一个测试 `.glb` 角色。
- 创建 `PetController.gd`。
- 默认播放 `idle`。
- 键盘或鼠标输入时触发 `wave`。
- 动作结束后回到 `idle`。
- 在右手添加 `BoneAttachment3D`。
- 给右手挂一个简单测试道具。

成功标准：

- 本地模型可以待机。
- 任意键盘或鼠标输入可以触发挥手。
- 手持道具能跟随手部动画移动。

### 阶段 2：动作系统

不要把动作逻辑全部硬编码在脚本里。建议做一个动作注册表。

动作数据示例：

```text
name: wave
animation: wave
priority: 10
interruptible: true
cooldown: 1.0
trigger: any_input
```

任务：

- 增加动作队列。
- 增加冷却时间。
- 增加优先级。
- 如果目标动画缺失，播放降级动画。
- 预留音效和粒子效果接口。

### 阶段 3：装扮系统

任务：

- 定义装扮槽位：右手、头部、背部。
- 定义装扮元数据。
- 加载装扮 GLB 场景。
- 把装扮挂到 `BoneAttachment3D`。
- 本地保存已装备的装扮。

### 阶段 4：模型下载和本地缓存

任务：

- 增加 HTTP 请求，创建生成任务。
- 增加生成状态轮询。
- 下载 `.glb`。
- 保存生成模型到用户目录。
- 下次启动时加载缓存模型。
- 增加失败和重试 UI。

### 阶段 5：接入 AI 后端

任务：

- Godot 客户端连接后端接口。
- 上传图片。
- 轮询任务状态。
- 下载生成好的 GLB。
- 在本地验证是否能加载。
- 保存模型元数据。

## 推荐里程碑

### 里程碑 1：离线原型

目标：不依赖 AI，先证明桌宠核心玩法成立。

交付物：

- 一个可用 GLB 角色。
- 待机动画。
- 输入触发的挥手动画。
- 一个手持道具。
- 本地装扮挂载。

### 里程碑 2：标准模型规范

目标：明确 AI 最终必须产出什么格式的模型。

交付物：

- 骨骼命名文档。
- 动画命名文档。
- 装扮挂点文档。
- 一个符合规范的测试 GLB 模型。

### 里程碑 3：后端原型

目标：通过 API 生成模型，同时不在游戏里暴露第三方密钥。

交付物：

- `POST /generate`
- `GET /generate/{task_id}`
- `.glb` 下载接口
- Meshy 或 Tripo 的服务测试

### 里程碑 4：验证管线

目标：防止损坏或不稳定模型进入游戏。

交付物：

- GLB 加载验证。
- 骨骼验证。
- 尺寸和朝向规范化。
- 基础动画测试。
- 友好的失败提示。

### 里程碑 5：游戏内 AI 生成

目标：让用户在游戏内完成上传和生成流程。

交付物：

- 上传图片 UI。
- 生成进度 UI。
- 生成模型下载。
- 生成模型本地缓存。
- 生成模型可以触发标准动作。

## 风险清单

### 风险：AI 生成模型拓扑不好

影响：  

- 手臂动画时拉伸或裂开。
- 关节塌陷。
- 道具挂载位置异常。

缓解：

- 使用 Q版/玩具风。
- 要求 A-Pose 或 T-Pose。
- 第一版只做小幅度动作。
- 验证动画变形。
- 失败时降级为整体动画。

### 风险：不同 AI 服务的骨骼不一致

影响：

- 你的动作库无法稳定复用。

缓解：

- 统一转换到标准骨骼。
- 做骨骼映射表。
- 无法映射的模型直接拒绝。
- 优先选择骨骼规范更稳定的服务商。

### 风险：API 成本太高

影响：

- 免费用户大量生成会带来成本压力。

缓解：

- 免费用户限制生成次数。
- 生成结果本地缓存。
- 增加付费生成次数。
- 后续允许高级用户填写自己的 API Key。

### 风险：本地 AI 生成太重

影响：

- 游戏安装包变大。
- 用户电脑配置要求高。
- 发行和售后复杂。

缓解：

- 早期 AI 生成放在云端。
- 普通桌宠玩法保持本地运行。
- 首次生成后把 GLB 缓存在本地。

## 技术建议

先围绕“标准角色系统”来做游戏，不要让某个 AI 服务商决定你的游戏架构。

正确顺序是：

```text
1. 先让一个标准 GLB 桌宠在 Godot 里跑起来。
2. 增加固定动作和输入触发。
3. 增加装扮挂点。
4. 定义模型规范。
5. 增加后端 AI 生成。
6. 把生成模型整理成你的标准规范。
7. 再把生成模型加载进已有桌宠系统。
```

这样即使以后更换 AI 服务商，游戏本体也不会被推倒重来。
